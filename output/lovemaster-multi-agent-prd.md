# PRD: Lovemaster 多 Agent 恋爱陪聊 / Manus 模式

> 版本：v0.1  
> 日期：2026-03-27

## 1. 产品目标

把 Lovemaster 从“普通恋爱聊天 + Coach 工具面板”升级成一个能处理聊天截图、能做问题重写、能按模式决定是否调工具的恋爱 AI 助手。

产品要同时满足两类需求：

### A. 陪聊型需求

- “他这样说是什么意思？”
- “我现在怎么回比较好？”
- “你帮我润色成温柔一点 / 高冷一点 / 带点调皮”

### B. 任务型需求

- “帮我整理一份追求计划”
- “根据这段聊天，帮我判断接下来三天应该怎么推进”
- “分析完后帮我生成一段长回复 / 备选话术 / 约会计划”

---

## 2. 目标用户

### 用户 1：高频回复焦虑型

- 有明确聊天对象
- 经常卡在“该怎么回”
- 希望上传截图后立即得到判断和回复建议

### 用户 2：情感分析依赖型

- 需要情绪陪伴
- 不只是想要一句回复，更想理解对方在想什么
- 关注关系推进、边界、红旗信号

### 用户 3：重度任务委托型

- 愿意让 AI 帮自己做更长链路的事情
- 接受 AI 搜索、整理、生成、规划
- 需要“过程可视化”和“结果可交付”

---

## 3. 产品定位

Lovemaster 不是通用聊天机器人，也不是单纯的撩妹话术机。

它的定位是：

- 在 Love 模式下，做“懂关系和情绪的恋爱顾问”
- 在 Coach / Manus 模式下，做“能规划和执行的恋爱任务助理”

---

## 4. 核心模式设计

## 4.1 Love 模式

定位：快反馈、强共情、低打扰。

默认行为：

- 接收文本或截图
- 自动做 OCR / 重写 / 关系分析
- 输出解释 + 建议 + 可直接发送的话术
- 默认不调用外部工具

适合场景：

- 临时回消息
- 解读聊天语气
- 优化回复措辞
- 做恋爱陪伴和情绪安抚

## 4.2 Coach / Manus 模式

定位：多步骤任务、过程可见、可调用工具。

默认行为：

- 同样支持文本或截图输入
- 先做 OCR / 重写 / 意图澄清
- 再由大脑 Agent 判断：仅回答，还是切换到工具执行
- 若进入执行态，过程在 ManusPanel 中展示

适合场景：

- 搜资料
- 做阶段性关系策略
- 输出长文案、计划、资料
- 调用已有工具与 MCP

---

## 5. 用户旅程

## 5.1 Love 模式旅程

1. 用户进入 Love 模式
2. 粘贴聊天记录或上传截图
3. 系统显示“正在识别截图 / 正在整理对话”
4. 系统识别出双方对话内容和上下文
5. 系统重写成一个更适合大脑 Agent 理解的问题
6. 大脑 Agent 输出：
   - 对方意图判断
   - 用户当前可采取的策略
   - 2 到 3 条不同语气的回复建议

## 5.2 Coach / Manus 模式旅程

1. 用户进入 Coach 模式
2. 输入问题或上传截图
3. Intake Agent 处理原始输入
4. Brain Agent 判断：
   - 直接给答案
   - 或发起任务执行
5. 若发起任务：
   - 面板展示任务进度
   - Tool Agent 逐步执行
   - 输出文件、资料或最终建议

---

## 6. 功能需求

## 6.1 输入层

- 支持文本输入
- 支持聊天截图上传
- 支持单图优先，后续可扩展多图
- 接收截图后给出明确状态反馈

### 验收标准

- 用户上传截图后，系统必须明确显示“已识别图片并开始分析”
- 识别失败时必须返回失败原因或补救提示

## 6.2 文字重写和识别 Agent

职责：

- OCR 提取截图内容
- 判断说话双方
- 尽量按时间线整理上下文
- 对用户原始提问做精细化 rewrite
- 输出结构化上下文给 Brain Agent

输出至少包括：

- 原始用户问题
- OCR 文本
- 整理后的对话摘要
- 说话人标签
- 不确定项
- 重写后的标准问题

## 6.3 大脑 Agent

职责：

- 决定回答风格
- 判断问题属于陪聊、分析、回复生成还是任务委托
- 决定是否调用工具代理
- 生成最终用户可见回复

Love 模式输出要求：

- 情绪确认
- 关系分析
- 回复建议
- 必要时给出风险提醒

Coach 模式输出要求：

- 若无需工具，直接回答
- 若需要工具，先给出目标理解，再进入任务执行

## 6.4 工具调用 Agent

职责：

- 只在 Coach / Manus 模式中按需启用
- 接收大脑 Agent 的任务目标
- 复用现有 Tool / MCP 能力
- 输出可展示的任务轨迹和结果

## 6.5 结果呈现

Love 模式：

- 以对话气泡呈现
- 重点突出“对方是什么意思”和“你可以怎么回”

Coach 模式：

- 对话区展示最终结论
- 面板区展示任务、终端、预览

## 6.6 安全与边界

- 不鼓励操控、PUA、骚扰、跟踪
- 遇到明显情感暴力、威胁、未成年人或违法内容时降级
- 对截图中的隐私内容做最小化处理
- 日志不可输出完整原图 URL、原始对话全文或 token 密钥

---

## 7. 非功能需求

### 响应速度

- Love 模式纯文本首屏反馈目标：2 秒内出现状态，8 秒内出现首个有效结论
- Love 模式带图首屏反馈目标：2 秒内出现状态，12 秒内出现首个有效结论
- Coach 模式长任务允许更长，但必须持续有状态更新

### 准确性

- OCR 无法确定的内容必须显式标记
- 不允许把模糊内容当成确定事实输出

### 可观测性

- 记录模式、是否带图、是否进入工具态、耗时、失败类型
- 不记录敏感正文原文到普通日志

### 可扩展性

- 第一阶段支持单图
- 后续可扩展多图、多轮修正、人工确认节点

---

## 8. 不做什么

第一阶段明确不做：

- 不做完整社交平台账号接管
- 不做自动代聊发送
- 不做多人关系图谱
- 不做全量框架升级到最新版 Graph Runtime
- 不做复杂的人设市场 / prompt marketplace

---

## 9. 成功指标

上线后重点关注：

- 带图会话占比
- Love 模式回复采纳率
- Coach 模式任务完成率
- 带图会话平均响应时长
- OCR 失败率
- 用户二次追问率

建议的首批业务指标：

- 带图输入占总会话 > 20%
- Love 模式首轮有效回复率 > 85%
- Coach 模式任务完成率 > 70%
- OCR 失败率 < 10%

---

## 10. 版本规划

## Phase 1

- 打通截图输入到模型
- 上线三 Agent 基础编排
- Love / Coach 两模式完成分层
- 面板展示基础过程

## Phase 2

- 增加多风格回复卡片
- 增加 OCR 纠错和人工确认
- 增加更清晰的策略模板
- 评估迁移到 Graph / Supervisor

## Phase 3

- 支持多图、多轮上下文拼接
- 引入长期偏好与用户画像
- 支持更复杂的并行子代理
